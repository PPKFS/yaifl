IDEAS FOR FUTURE: Precondition Builder DSL

Current Problem:
- Preconditions like `theObjectWhenSwitchedOn` are ugly because they require:
  1. Manual entity ID comparison
  2. Manual type checking (getDeviceMaybe)
  3. Complex nested Maybe operations
  4. Hard to compose multiple conditions

Proposed Solution:
Create a Precondition Builder DSL that provides:

1. Monadic interface for building preconditions step-by-step
2. Helper functions for common checks:
   - getMainObject, getSecondaryObject
   - isKind, hasProperty
   - isSwitchedOn, isOpen, isLocked
   - isCarriedBy, isInRoom
   - entity comparison helpers

3. Clean composition operators:
   - AND (&&.) and OR (||.) combinators
   - negation (notP)
   - sequential checks with early termination

4. Better error handling and debugging:
   - Clear precondition names
   - Better failure messages
   - Stackable conditions

Example Usage:
Instead of:
```haskell
theObjectWhenSwitchedOn th = Precondition (pure "the object when switched on") $ \args -> do
  let (mbT :: Maybe (Thing wm)) = args ^? #variables % argsMainObjectMaybe
  let d = mbT >>= getDeviceMaybe
  return $ Just (getID th) == (getID <$> mbT) && Just True == (switchedOn <$> d)
```

We could write:
```haskell
theObjectWhenSwitchedOn th = buildPrecondition "object is switched on" $ \args -> do
  actualThing <- getMainObject args
  guard (getID th == getID actualThing)
  device <- getDevice actualThing
  guard (device ^. #switchedOn)
  return True
```

Benefits:
- More readable and maintainable
- Less error-prone
- Easier to compose complex conditions
- Better separation of concerns
- Reusable components

This would significantly clean up action preconditions while maintaining all current functionality.
